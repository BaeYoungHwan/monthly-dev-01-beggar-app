| 상태 | 경로 |
|------|------|

---

## 상태 표시

| 기호 | 의미 |
|------|------|
| `[ ]` | 미시작 |
| `[🔄]` | 진행 중 |
| `[x]` | 완료 |

---

## 표준 작업 흐름

### 1. 작업 시작 전
- `docs/exec-plans/active/`에 Phase별 작업 목록 파일 생성
- 파일명 예) `phase1-auth.md`, `phase2-dashboard.md`

### 2. 작업 중
- 현재 진행 항목: `[ ]` → `[🔄]`
- 컨텍스트 단절 대비: `[🔄]` 상태로 저장하면 재시작 시 이어서 진행 가능

### 3. 항목 완료 시
- `[🔄]` → `[x]` 표시
- `/commit` 실행

### 4. Phase 전체 완료 시
- `active/` 파일 → `completed/`로 이동
- `docs/QUALITY_SCORE.md` 업데이트

---

## 컨텍스트 재시작 ("다음 작업 하자")

1. `docs/ref/session-state.md` 읽기 → git 상태 파악
2. `docs/exec-plans/active/` 읽기 → 현재 작업 목록 확인
3. `[🔄]` 항목부터 이어서 진행
4. `[🔄]` 항목 없으면 `[ ]` 첫 번째 항목 시작

---

## 실행 계획 파일 형식 예시

```markdown
# Phase 1 — 인증 시스템

## 목표
JWT 기반 로그인/회원가입 구현

## 작업 목록

- [x] User 모델 설계
- [x] 회원가입 API
- [🔄] 로그인 API (JWT 발급)
- [ ] 토큰 갱신 API
- [ ] 미들웨어 (인증 검증)

## 완료 기준
- 로그인 후 보호된 엔드포인트 접근 가능
- 토큰 만료 시 자동 갱신
```

---

## 주의사항

- 작업 중단 시 반드시 `[🔄]`로 상태 업데이트 후 종료
- 한 번에 여러 항목을 `[🔄]`로 표시하지 말 것 (현재 항목 1개만)
- `/commit`은 `[x]` 전환 직후 실행하는 것이 원칙
