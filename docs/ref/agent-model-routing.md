# 에이전트 모델 라우팅

작업 유형별 최적 모델 선택 기준 (Delegation Enforcer 패턴).

---

## 기본 원칙

비싼 모델을 기본값으로 쓰지 않는다.
작업의 복잡도에 맞는 모델을 선택해서 **토큰 비용과 속도를 최적화**한다.

---

## 모델별 적용 범위

### Haiku — 빠른 탐색
**언제**: 정보 수집, 검색, 파일 읽기, 간단한 분류
```
- 파일 내용 grep / 구조 파악
- 로그 분석 (패턴 찾기)
- 문서 요약 (짧은 텍스트)
- 코드 존재 여부 확인
```

### Sonnet — 실질 개발
**언제**: 코딩, 디버깅, 리팩터링, 테스트 작성
```
- 기능 구현 (함수/클래스/모듈)
- 버그 수정
- 코드 리뷰 (기술적 피드백)
- API 연동 구현
- 테스트 코드 작성
```

### Opus — 설계와 판단
**언제**: 복잡한 아키텍처 결정, Plan 모드, 트레이드오프 분석
```
- 시스템 아키텍처 설계
- 기술 스택 선택
- PRD 검토 및 Phase 분할
- 보안 취약점 분석
- 복잡한 리팩터링 전략
```

---

## 에이전트 라우팅 규칙

```
탐색 서브태스크 → model: haiku
개발 서브태스크 → model: sonnet
설계 서브태스크 → model: opus
```
> `subagent_type`은 에이전트 역할("Explore", "Plan", "general-purpose")을 지정합니다.
> 모델 선택은 별도의 `model:` 파라미터로 지정합니다.

병렬 에이전트 사용 시 각 에이전트의 `.md` 파일에 `model:` 프런트매터 명시:

```yaml
---
name: frontend-agent
model: sonnet   # haiku / sonnet / opus
---
```

---

## 실전 예시

| 작업 | 모델 | 이유 |
|------|------|------|
| 로그 파일에서 에러 패턴 찾기 | Haiku | 단순 grep 수준 |
| React 컴포넌트 구현 | Sonnet | 중간 복잡도 코딩 |
| 전체 DB 스키마 재설계 | Opus | 아키텍처 판단 필요 |
| 단위 테스트 작성 | Sonnet | 구현 수준 작업 |
| 마이크로서비스 분리 전략 | Opus | 시스템 설계 |
| 파일 존재 여부 확인 | Haiku | 탐색만 필요 |
