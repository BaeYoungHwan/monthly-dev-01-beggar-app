# 검증 프로토콜

작업 크기별 검증 전략입니다.

---

## 검증 레벨

### Level 1 — 함수 단위
**기준**: 변경 범위가 함수 1~3개 이내
```
1. 단위 테스트 통과 확인
2. 타입 에러 없음 확인
3. 커밋
```

### Level 2 — 모듈 단위
**기준**: 파일 3개 이상 또는 API 엔드포인트 변경
```
1. 단위 테스트 + 통합 테스트 통과
2. 관련 모듈 회귀 테스트 실행
3. 로컬 서버 실행 후 수동 확인
4. 커밋
```

### Level 3 — Phase 완료
**기준**: Phase 전체 작업 완료
```
1. 전체 테스트 스위트 pass^3
2. E2E 시나리오 수동 검증
3. docs/QUALITY_SCORE.md 업데이트
4. exec-plans/active/ → completed/ 이동
5. 커밋
```

### Level 4 — 릴리즈 전
**기준**: 프로덕션 배포 전
```
1. 전체 테스트 pass^5
2. /security-review 실행
3. docs/RELIABILITY.md 업데이트
4. 스테이징 환경 검증
5. 태그 생성
```

---

## Ralph 루프 (`/ralph` 스킬)

자동 검증이 필요할 때:
```
plan → exec → verify → fix → verify → ... → success
```
실패 시 자동으로 수정 후 재검증. 수동 개입 없이 완료 보장.

---

## 검증 실패 처리

- **circuit-breaker**: 동일 에러 3회 반복 시 자동 중단 → `logs/circuit-breaker.log`
- 중단 후: 에러 로그 분석 → 접근법 변경 → 재시작

---

## 주의사항

- "작동하는 것 같다"는 검증이 아님 — 테스트로 증명할 것
- 빠른 커밋 압박에도 레벨에 맞는 검증 완료 후 커밋
- 검증 건너뛰기는 `Not-done:` Trailer에 명시
